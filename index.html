🚀 Transformer - Der komplette Kurs

Von RNNs zu Self-Attention: Verstehe die Architektur der modernen KI

🚀 Was sind Transformer?

Transformer sind eine Architektur für neuronale Netze, die 2017 in dem Papier "Attention is All You Need" vorgestellt wurde. Sie haben die Welt der künstlichen Intelligenz revolutioniert.

Kernidee: Statt Daten sequenziell zu verarbeiten (wie RNNs), verarbeiten Transformer alle Eingaben parallel und nutzen einen Mechanismus namens "Self-Attention", um zu verstehen, welche Teile des Inputs zusammenhängen.

Warum sind Transformers so mächtig?

  • Parallelisierung: Alle Tokens können gleichzeitig verarbeitet werden → schneller Training
  • Lange Abhängigkeiten: Self-Attention sieht jedes Token im Kontext aller anderen Tokens
  • Skalierbarkeit: Kann auf riesige Datenmengen trainiert werden (GPT, BERT, etc.)
  • Effizienz: Weniger Speicher und Rechenzeit als RNNs für lange Sequenzen
✨ Fun Fact: ChatGPT, Claude, Gemini, Llama – alle sind Transformer-basiert! Die Modelle, mit denen du heute sprichst, basieren auf dieser Architektur.

Aufbau

Ein Transformer besteht aus:

  1. Embedding Layer: Wandelt Wörter in Vektoren um
  2. Positional Encoding: Fügt Positions-Info hinzu
  3. Encoder Stack: N x (Multi-Head Attention + Feed-Forward)
  4. Decoder Stack: N x (Masked Attention + Encoder-Decoder Attention + Feed-Forward)
  5. Output Layer: Gibt Wahrscheinlichkeiten für nächstes Token

1️⃣ RNNs vs. Transformers

Um Transformers wirklich zu verstehen, schauen wir uns an, wie sie sich von RNNs unterscheiden.

Recurrent Neural Networks (RNNs)

RNNs verarbeiten Daten sequenziell – Token für Token:

Token 1 → Hidden State 1 → Token 2 → Hidden State 2 → Token 3 → ...

Vorteile:

  • Speichert Zustand aus vorherigen Tokens
  • Gut für Variable-Länge Sequenzen
  • Vergleichsweise einfach zu verstehen

Nachteile:

  • ❌ Keine Parallelisierung möglich
  • ❌ Langsames Training (muss warten auf vorherige Token)
  • ❌ Schwierig, lange Abhängigkeiten zu lernen (Vanishing Gradient)
  • ❌ Für lange Texte langsam und speicherintensiv

Transformers

Transformers verarbeiten Daten parallel – alle Tokens gleichzeitig:

[Token 1, Token 2, Token 3, ...] → Self-Attention → [Output 1, Output 2, Output 3, ...]

Vorteile:

  • ✅ Vollständige Parallelisierung
  • ✅ Schnelles Training auf modernen GPUs
  • ✅ Sieht alle Abhängigkeiten auf einmal
  • ✅ Skaliert gut auf sehr lange Sequenzen

Nachteile:

  • Mehr Speicher benötigt (aller Tokens gleichzeitig)
  • Komplexer zu verstehen und zu implementieren
  • Längere Sequenzen = quadratische Speicherkomplexität
Feature RNN Transformer
Parallelisierung Nein ❌ Ja ✅
Training-Geschwindigkeit Langsam Schnell
Lange Abhängigkeiten Schwierig Einfach
Speicher (lange Sequenzen) Effizient O(n²)
State Propagation Sequential Parallel
❓ Warum sind Transformers schneller zu trainieren als RNNs?
A) Weil sie Tokens parallel verarbeiten, nicht sequenziell
B) Weil sie weniger Parameter haben
C) Weil sie keine Hidden States brauchen
D) Weil Self-Attention magisch ist
✅ Richtig! RNNs müssen Token 1 verarbeiten, dann Token 2, dann Token 3, etc. Transformers können alle gleichzeitig verarbeiten, was auf GPUs viel schneller ist.

2️⃣ Embeddings & Input Preparation

Bevor ein Transformer mit Text arbeiten kann, muss dieser in Zahlen umgewandelt werden. Das ist das Embedding!

Was ist ein Embedding?

Ein Embedding ist eine dichte Vektorrepräsentation eines Wortes. Statt ein Wort als String zu speichern, speichern wir es als Vektor von Zahlen:

Wort "König"
Embedding: [0.2, -0.1, 0.8, ..., 0.3] (typisch 512 oder 4096 Dimensionen)

Diese Zahlen sind gelernt! Das Netzwerk lernt, Wörter mit ähnlicher Bedeutung ähnliche Embeddings zu geben.

Embedding-Eigenschaften

  • Semantik: Ähnliche Wörter haben ähnliche Vektoren
  • Algebraische Operationen: "König" - "Mann" + "Frau" ≈ "Königin"
  • Dimension: Häufig 512 (BERT, GPT-2) bis 4096 (GPT-4)
  • Lernbar: Die Embeddings werden während des Trainings optimiert

Das volle Input-Preparation-Rezept

Ein Transformer braucht zwei Sachen:

  1. Token Embeddings: Die Bedeutung des Wortes
  2. Positional Embeddings: Die Position im Satz
Input Processing Pipeline:

"Ich bin ein Transformer"
↓
[101, 12, 15, 345, 123] (Token IDs)
↓
Embedding Lookup
↓
[[0.2, ..., 0.3], [0.5, ..., -0.1], ...] (512-dim Vektoren)
↓
+ Positional Encodings
↓
Ready für Transformer! ✅

Tokenization

Bevor Embeddings verwendet werden, muss Text in "Tokens" aufgeteilt werden. Das kann sein:

  • Character-level: "Hallo" → ['H', 'a', 'l', 'l', 'o']
  • Word-level: "Hallo Welt" → ['Hallo', 'Welt']
  • Subword-level (BPE): "Tokenization" → ['Token', 'ization'] (Modern!)

GPT-2, GPT-3, GPT-4 verwenden Byte Pair Encoding (BPE) – das ist ein Mittelweg zwischen Wort- und Character-Level.

❓ Was ist das Hauptziel eines Embeddings?
A) Text in dichte numerische Vektoren zu konvertieren
B) Text zu komprimieren
C) Text zu verschlüsseln
D) Text auszugeben
✅ Richtig! Embeddings wandeln kategorische Text-Daten in kontinuierliche numerische Vektoren um, die ein neuronales Netz verarbeiten kann.

3️⃣ Self-Attention Mechanismus

Self-Attention ist das Herzstück des Transformers. Es ist elegant, mächtig und der Grund, warum Transformers so gut funktionieren.

Das Problem: "Welche Tokens sind relevant?"

Betrachte den Satz: "Das Buch auf dem Tisch ist blau."

Wenn wir "Buch" verarbeiten, müssen wir wissen, dass "Das" und "auf" und "Tisch" und "ist blau" relevant sind. Aber wie?

Self-Attention Antwort:
Jedes Token schaut sich alle anderen Tokens an und entscheidet, wie relevant sie sind. Das geschieht durch Abfragen und Vergleichen.

Die 3 Komponenten: Q, K, V

Für Self-Attention brauchen wir drei Dinge pro Token:

  • Query (Q): "Was suche ich? Was bin ich interessiert?"
  • Key (K): "Bin ich relevant für die aktuelle Query?"
  • Value (V): "Wenn ich relevant bin, hier ist die Info, die ich weitergebe"
Analoie: Stell dir vor, jedes Token ist ein Student in einer Klasse. Query = "Ich will eine Antwort zu Thema X". Key = "Ich bin Experte in Thema X". Value = "Hier ist die Antwort".

Der Attention-Prozess (Schritt für Schritt)

  1. Berechne Q, K, V: Multipliziere Input-Embedding mit lernbaren Gewichtsmatrizen W_Q, W_K, W_V
  2. Berechne Scores: Dot-Product zwischen Q und K: score = Q · K^T
  3. Skaliere: Teile durch √d_k (verhindert zu extreme Werte)
  4. Softmax: Konvertiere zu Wahrscheinlichkeiten (0-1)
  5. Gewichte Values: Multipliziere Softmax-Gewichte mit V
Attention(Q, K, V) = softmax(Q·K^T / √d_k) · V

Beispiel: "Das Buch ist blau"

Self-Attention für Token "Buch":

Query(Buch): [0.1, 0.5, -0.3, ...] → "Ich suche relevante Wörter"

Key(Das): [0.2, 0.1, 0.2, ...] → Relevanz 0.8
Key(Buch): [0.1, 0.5, -0.2, ...] → Relevanz 1.0 (Ich selbst!)
Key(ist): [0.3, -0.2, 0.1, ...] → Relevanz 0.6
Key(blau): [0.05, 0.6, -0.5, ...] → Relevanz 0.7

Output: 0.8 × Value(Das) + 1.0 × Value(Buch) + 0.6 × Value(ist) + 0.7 × Value(blau)

Warum ist das genial?

  • ✅ Jedes Token berücksichtigt alle anderen Tokens
  • ✅ Die Gewichte sind datenabhängig (nicht fix!)
  • ✅ Lange Abhängigkeiten sind kein Problem
  • ✅ Leicht zu parallelisieren
❓ Was macht Q (Query) in Self-Attention?
A) Repräsentiert, was das aktuelle Token sucht
B) Speichert Informationen aus anderen Tokens
C) Berechnet Softmax-Gewichte
D) Ist nur ein zufälliger Vektor
✅ Richtig! Query sagt "Ich bin Token X und ich suche nach Information zum Thema ...". Key antwortet "Ich bin Token Y und ich habe Information zum Thema ...".

4️⃣ Scaled Dot-Product Attention

Jetzt gehen wir tiefer in die Mathematik und implementieren Self-Attention konkret.

Die Formel

Attention(Q, K, V) = softmax(QK^T / √d_k) V

Lass uns das auseinandernehmen:

Schritt 1: Q · K^T (Kompatibilität)

Multipliziere Query-Matrix mit transponierter Key-Matrix:

Q: (seq_len, d_k) → [0.1, 0.5, -0.3] K: (seq_len, d_k) → [0.2, 0.1, 0.2] Q · K^T: (seq_len, seq_len) → [[0.1×0.2 + 0.5×0.1 + (-0.3)×0.2], ...] = [[0.02 + 0.05 - 0.06], ...] = [[0.01], ...]

Das Ergebnis ist eine Attention-Matrix: Eine seq_len × seq_len Matrix, die zeigt, wie relevant jedes Token für jedes andere ist.

Schritt 2: Skalierung durch √d_k

Warum teilen wir durch √d_k?

Wenn d_k groß ist (z.B. 64), dann sind die Dot-Products sehr groß oder sehr klein. Das macht Softmax hart und unstabil. √d_k = √64 = 8 normalisiert die Werte.
scaled_scores = scores / sqrt(d_k) Beispiel: d_k = 64 scores = [12.3, -5.1, 8.7] scaled_scores = [12.3/8, -5.1/8, 8.7/8] = [1.54, -0.64, 1.09] ← Vernünftig!

Schritt 3: Softmax

Konvertiere Scores zu Wahrscheinlichkeiten:

attention_weights = softmax(scaled_scores) softmax([1.54, -0.64, 1.09]) = [e^1.54, e^-0.64, e^1.09] / (e^1.54 + e^-0.64 + e^1.09) = [4.66, 0.53, 2.97] / 8.16 = [0.57, 0.06, 0.37] ← Summe = 1.0 ✅

Jetzt haben wir eine Verteilung! Token 0 bekommt 57% Gewicht, Token 1 bekommt 6%, Token 2 bekommt 37%.

Schritt 4: Mit Values multiplizieren

Gewichte die Value-Vektoren und addiere sie:

output = attention_weights · V attention_weights: [0.57, 0.06, 0.37] V = [[1.0, 0.2, -0.5], [0.3, 0.8, 0.1], [0.5, 0.1, 0.7]] output = 0.57 × [1.0, 0.2, -0.5] + 0.06 × [0.3, 0.8, 0.1] + 0.37 × [0.5, 0.1, 0.7] = [0.57, 0.11, -0.28] + [0.02, 0.05, 0.01] + [0.19, 0.04, 0.26] = [0.78, 0.20, -0.01] ← Finales Ausgabe-Embedding!

Komplettes Bild

Schritt Input Shape Ausgabe Shape Zweck
Q · K^T (seq_len, d_k) × (d_k, seq_len) (seq_len, seq_len) Kompatibilität messen
/ √d_k (seq_len, seq_len) (seq_len, seq_len) Werte skalieren
softmax (seq_len, seq_len) (seq_len, seq_len) In Wahrscheinlichkeiten konvertieren
× V (seq_len, seq_len) × (seq_len, d_v) (seq_len, d_v) Values gewichten und aggregieren
⚠️ Speicherkomplexität: Die Attention-Matrix ist seq_len × seq_len. Bei einer 4000-Token-Sequenz sind das 16 Millionen Einträge! Das ist ein großes Problem bei sehr langen Texten.
❓ Warum skalieren wir durch √d_k und nicht einfach die Rohscores verwenden?
A) Um zu verhindern, dass Softmax-Gradienten zu klein werden
B) Um Speicher zu sparen
C) Um die Berechnung zu beschleunigen
D) Es ist nur eine Konvention
✅ Richtig! Große Scores führen zu sehr kleinen Gradienten in Softmax (Saturierung). √d_k normalisiert die Scores auf eine vernünftige Größe.

5️⃣ Multi-Head Attention

Eine Single-Head Attention ist gut, aber eine Multi-Head Attention ist besser! Das ist der Grund, warum Transformers "Multi-Head Attention" heißen.

Das Problem mit einem Head

Mit nur einer Attention-Funktion kann das Modell nur eine "Art" von Relationen erfassen. Stell dir vor, ein Sprachmodell könnte nur eine Regel pro Layer lernen – das wäre zu limitiert!

Lösung: Statt 1 Attention-Funktion, verwende h verschiedene Attention-Funktionen (Heads) parallel. Jeder Head kann sich auf unterschiedliche Relationen spezialisieren.

Wie Multi-Head Attention funktioniert

Für h Heads (typisch h=8) teile d (typisch d=512) in h kleinere Dimensionen (512/8 = 64 pro Head):

  1. Linear projections: Projiziere Q, K, V in h verschiedene Räume
  2. Parallele Attention: Berechne Attention für jeden Head unabhängig
  3. Concatenate: Verbinde alle Heads wieder
  4. Linear projection: Lineare Projektion der kombinierten Ausgabe
MultiHead(Q, K, V) = Concat(head_1, ..., head_h) W^O
wo head_i = Attention(Q W_i^Q, K W_i^K, V W_i^V)

Was spezialisiert sich jeder Head auf?

Das Modell lernt automatisch, was jeder Head machen soll:

Head 1: Fokus auf Subjekt-Verb-Objekt Struktur

Head 2: Fokus auf Adjektiv-Nomen Paare

Head 3: Fokus auf Referenzen (z.B. "er", "sie" → das ursprüngliche Subjekt)

Head 4: Fokus auf Position-Information

Head 5-8: Verschiedene Kombinationen von oben

Multi-Head Attention in PyTorch (Pseudocode)

class MultiHeadAttention(nn.Module): def __init__(self, d_model, num_heads): self.num_heads = num_heads self.d_k = d_model // num_heads # Lineare Projektionen für Q, K, V self.W_q = nn.Linear(d_model, d_model) self.W_k = nn.Linear(d_model, d_model) self.W_v = nn.Linear(d_model, d_model) self.W_o = nn.Linear(d_model, d_model) def forward(self, Q, K, V): # Projiziere Q = self.W_q(Q) # (batch, seq_len, d_model) K = self.W_k(K) V = self.W_v(V) # Split in multiple heads Q = Q.view(batch, seq_len, self.num_heads, self.d_k).transpose(1, 2) # Jetzt: (batch, num_heads, seq_len, d_k) # Berechne Attention für jeden Head scores = Q @ K.transpose(-2, -1) / sqrt(self.d_k) weights = softmax(scores, dim=-1) output = weights @ V # (batch, num_heads, seq_len, d_k) # Concatenate Heads output = output.transpose(1, 2).contiguous() output = output.view(batch, seq_len, d_model) # Final linear projection return self.W_o(output)

Beispiel: 8 Heads, d=512

Größe Wert Form
d_model 512 Gesamte Embedding-Dimension
num_heads 8 Anzahl paralleler Attention-Funktionen
d_k pro Head 64 512 / 8 = 64
Q pro Head (seq_len, 64) Projektion in 64-dim Raum
Attention-Matrix (seq_len, seq_len) Wie bei Single-Head Attention
✨ Vorteil: Mit 8 Heads bekommt das Modell 8 verschiedene "Perspektiven" auf die Daten. Das macht es viel ausdrucksstärker!
❓ Wenn d_model=512 und num_heads=8, wie groß ist d_k pro Head?
A) 64
B) 512
C) 4096
D) 8
✅ Richtig! d_k = d_model / num_heads = 512 / 8 = 64. Die Gesamtdimension wird gleichmäßig auf alle Heads verteilt.

6️⃣ Positional Encoding

Hier ist ein Geheimnis: Self-Attention hat keine inhärente Ahnung von Reihenfolge! Wenn wir die Token shuffeln, gibt es dasselbe Ergebnis. Das ist ein großes Problem für eine Sprache, die sequenziell ist!

Das Problem

Diese beiden Sätze würden für einen Transformer ohne Positional Encoding identisch sein:

Satz 1: "Hund sah Katze"
Satz 2: "Katze sah Hund"

Ohne Position-Info: Gleiche Tokens, gleiche Attention-Gewichte, gleicher Output!

Die Lösung: Positional Encoding

Für jede Position pos in der Sequenz und jede Dimension d_model, definieren wir:

PE(pos, 2i) = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

Das sieht kompliziert aus, aber das Wichtige ist: Jede Position bekommt einen einzigartigen Vektor!

Ein konkretes Beispiel

Sei d_model=4 (klein für Illustration). Positionen 0-3:

PE[0] = [sin(0), cos(0), sin(0), cos(0)] = [0, 1, 0, 1] PE[1] = [sin(1), cos(1), sin(1/100), cos(1/100)] = [0.841, 0.540, 0.010, 1.000] PE[2] = [sin(2), cos(2), sin(2/100), cos(2/100)] = [0.909, -0.416, 0.020, 0.999] PE[3] = [sin(3), cos(3), sin(3/100), cos(3/100)] = [0.141, -0.990, 0.030, 0.999]

Wichtig: Diese PE-Vektoren werden zu den Embeddings addiert!

Token-Embedding + Positional Encoding:

x_0 = embedding("Hund") + PE[0]
x_1 = embedding("sah") + PE[1]
x_2 = embedding("Katze") + PE[2]

Jedes Token hat jetzt seine Position codiert!

Warum Sin/Cos?

Die Wahl von Sinus und Kosinus ist clever:

  • Periodisch: Sin/Cos wiederholen sich, was Linearität erlaubt ("Position 10 relativ zu Position 5 ist die Differenz von Position 15 relativ zu Position 10")
  • Verschiedene Frequenzen: Höhere Dimensionen haben niedrigere Frequenzen (mehr Wiederholungen), niedrigere Dimensionen haben höhere Frequenzen
  • Begrenzt: Sin/Cos sind immer zwischen -1 und 1, also stabil
  • Keine neuen Gewichte zu lernen: PE ist festgelegt, nicht trainiert

Alternative: Learnable Positional Embeddings

BERT verwendet stattdessen lernbare Positional Embeddings:

position_embeddings = nn.Embedding(max_position, d_model) # Während Training: Die Position-Embeddings werden wie andere Parameter gelernt output = token_embedding + position_embeddings[position]

Beide Ansätze funktionieren! Die sin/cos Variante ist einfacher, learnable Embeddings könnten theoretisch mehr Flexibilität geben.

❓ Warum braucht ein Transformer Positional Encoding?
A) Weil Self-Attention von Natur aus keine Reihenfolge versteht
B) Um Speicher zu sparen
C) Um Tokens zu komprimieren
D) Um die Berechnung zu beschleunigen
✅ Richtig! Self-Attention ist permutations-invariant – es schaut nur auf Attention-Gewichte, nicht auf die Reihenfolge. Positional Encoding fügt Position-Information hinzu.

7️⃣ Der Transformer Block

Ein einzelner Transformer "Block" ist das Grundgerüst des gesamten Transformers. Ein voller Transformer besteht aus vielen dieser Blöcke übereinander gestapelt.

Was ist in einem Block?

Ein Transformer Block hat diese Komponenten in dieser Reihenfolge:

  1. Multi-Head Self-Attention
  2. Add & Layer Normalization (Residual Connection)
  3. Feed-Forward Network (FFN)
  4. Add & Layer Normalization (Residual Connection)

Residual Connections (Skip Connections)

Dieser Teil ist wichtig! Nach jeder Subschicht addieren wir die Eingabe wieder:

output = LayerNorm(input + SubLayer(input))

(Statt nur: output = SubLayer(input))

Warum? Das verhindert das "Vanishing Gradient Problem" in tiefen Netzen!

Gradient Flow: Mit Residual Connections können Gradienten direkt durch die Skip Connection zurückfließen, auch wenn die Subschicht Gradienten auf 0 schrumpfen würde.

Layer Normalization

Nach jeder Subschicht normalisieren wir die Ausgabe:

LayerNorm(x) = gamma * (x - mean(x)) / sqrt(var(x) + epsilon) + beta Wo: - mean(x), var(x): Statistiken pro Sample (nicht über Batch!) - gamma, beta: Lernbare Parameter

Das stabilisiert das Training und hilft dem Modell, schneller zu konvergieren.

Feed-Forward Network (FFN)

Nach Attention kommt ein einfaches, aber mächtiges Zwei-Layer MLP:

FFN(x) = max(0, x W_1 + b_1) W_2 + b_2

Oder mit GELU (neueres aktivierter):

FFN(x) = GELU(x W_1 + b_1) W_2 + b_2
class FeedForward(nn.Module): def forward(self, x): # Layer 1: d_model → d_ff (4 * d_model ist Standard) x = F.relu(self.linear1(x)) # Dropout (optional) x = self.dropout(x) # Layer 2: d_ff → d_model x = self.linear2(x) return x

Visualisierung eines Blocks

↓
+ Input (Residual) → (seq_len, d_model)
↓

Vollständiger Block Code

class TransformerBlock(nn.Module): def __init__(self, d_model, num_heads, d_ff, dropout=0.1): self.attention = MultiHeadAttention(d_model, num_heads) self.ffn = FeedForward(d_model, d_ff) self.norm1 = nn.LayerNorm(d_model) self.norm2 = nn.LayerNorm(d_model) self.dropout1 = nn.Dropout(dropout) self.dropout2 = nn.Dropout(dropout) def forward(self, x): # Attention + Residual + Norm attn_output = self.attention(x, x, x) x = self.norm1(x + self.dropout1(attn_output)) # FFN + Residual + Norm ffn_output = self.ffn(x) x = self.norm2(x + self.dropout2(ffn_output)) return x

8️⃣ Die komplette Transformer Architektur

Jetzt kombinieren wir alles: Das ist der "Attention is All You Need" Transformer aus 2017.

Encoder-Decoder Architektur

Der ursprüngliche Transformer hat zwei Teile:

  1. Encoder: Liest die ganze Input-Sequenz
  2. Decoder: Erzeugt die Output-Sequenz Token für Token

Encoder

Der Encoder nimmt eine Input-Sequenz und kodiert sie in eine Kontextrepräsentation:

Encoder Pipeline:

Input Text: "Ich bin ein Transformer"
↓
Tokenization + Embeddings + Positional Encoding
↓
[x_1, x_2, x_3, x_4] (512-dim vectors)
↓
N × Transformer Block (Self-Attention + FFN)
(6 Blöcke in der Base Version)
↓
Encoder Output: [z_1, z_2, z_3, z_4]
(Kontextrepräsentationen für jeden Token)

Decoder

Der Decoder erzeugt die Ausgabe Token für Token, dabei:

  • Schaut auf die bisherige Output-Sequenz (Masked Self-Attention)
  • Schaut auf den Encoder-Output (Cross-Attention)
  • Erzeugt ein neues Token
Decoder Pipeline (Autoregressive):

Start: [START]
↓
Embedding + Positional Encoding
↓
Masked Self-Attention (kann nur bisherige Tokens sehen)
↓
Cross-Attention mit Encoder Output
↓
FFN
↓
Output Layer + Softmax
↓
Token 1 (z.B. "I")
↓
Wiederhole mit [START, I] → Token 2 (z.B. "am")
Wiederhole mit [START, I, am] → Token 3 (z.B. "a")
... etc

Masked Self-Attention im Decoder

Im Decoder können wir nicht in die Zukunft schauen! Deshalb "maskieren" wir zukünftige Positionen:

# Normale Attention-Scores scores = Q @ K.T # (seq_len, seq_len) # Masken wir zukünftige Positionen mask = torch.triu(torch.ones(seq_len, seq_len), diagonal=1).bool() scores = scores.masked_fill(mask, float('-inf')) # Softmax: -inf wird zu 0 Wahrscheinlichkeit weights = softmax(scores, dim=-1)

Das erzeugt eine untere Dreiecksmatrix in der Attention:

Masked Attention (4 Token):

[ A 0 0 0 ]
[ A A 0 0 ]
[ A A A 0 ]
[ A A A A ]

Token 3 kann nur Tokens 1, 2, 3 sehen, nicht Token 4!

Cross-Attention (Encoder-Decoder Attention)

Dies ist wichtig: Der Decoder schaut mit seinen Queries auf die Encoder-Outputs als Keys und Values:

# Decoder Self-Attention output = attention(Q=decoder_output, K=decoder_output, V=decoder_output) # Cross-Attention output = attention(Q=decoder_output, K=encoder_output, V=encoder_output)

So kann der Decoder "fragen" "Welche Teile des Inputs sind relevant für mein nächstes Token?"

Output Layer und Softmax

Ganz am Ende:

# Decoder Output: (seq_len, d_model) logits = linear_layer(decoder_output) # (seq_len, vocab_size) # Wahrscheinlichkeiten probs = softmax(logits, dim=-1) # (seq_len, vocab_size) # Nächstes Token: argmax oder sampling next_token = argmax(probs[-1]) # Index des wahrscheinlichsten Tokens

Training vs. Inference

Aspekt Training Inference
Input Ganze Input + Target-Sequenz Input + 1 Token zur Zeit
Geschwindigkeit Parallel über alle Positionen Sequenziell, 1 Token/Forward Pass
Masked Attention Ja (Teacher Forcing) Ja (nur 1 Token sichtbar)
Speicher Hoch (ganze Sequenz) Niedrig (+ KV-Cache)
❓ Was ist Cross-Attention im Decoder?
A) Attention von Decoder zu Encoder Output
B) Attention nur über Tokens im Decoder
C) Attention über mehrere Sprachen hinweg
D) Etwas anderes
✅ Richtig! Cross-Attention ermöglicht es dem Decoder, auf den Encoder-Output zu schauen. Die Decoder-Ausgabe ist Query, der Encoder-Output ist Key und Value.

Multi-Head Attention (Vollständig)

import torch import torch.nn as nn import math class MultiHeadAttention(nn.Module): def __init__(self, d_model, num_heads): super().__init__() assert d_model % num_heads == 0 self.d_model = d_model self.num_heads = num_heads self.d_k = d_model // num_heads self.W_q = nn.Linear(d_model, d_model) self.W_k = nn.Linear(d_model, d_model) self.W_v = nn.Linear(d_model, d_model) self.W_o = nn.Linear(d_model, d_model) def forward(self, Q, K, V, mask=None): batch_size = Q.shape[0] # Linear transformations Q = self.W_q(Q) # (batch, seq_len, d_model) K = self.W_k(K) V = self.W_v(V) # Split into heads Q = Q.view(batch_size, -1, self.num_heads, self.d_k).transpose(1, 2) # (batch, num_heads, seq_len, d_k) K = K.view(batch_size, -1, self.num_heads, self.d_k).transpose(1, 2) V = V.view(batch_size, -1, self.num_heads, self.d_k).transpose(1, 2) # Scaled dot-product attention scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) weights = torch.softmax(scores, dim=-1) context = torch.matmul(weights, V) # Concat heads context = context.transpose(1, 2).contiguous() context = context.view(batch_size, -1, self.d_model) # Output projection output = self.W_o(context) return output

Positional Encoding

class PositionalEncoding(nn.Module): def __init__(self, d_model, max_len=5000): super().__init__() pe = torch.zeros(max_len, d_model) position = torch.arange(0, max_len, dtype=torch.float).unsqueeze(1) div_term = torch.exp(torch.arange(0, d_model, 2).float() * (-math.log(10000.0) / d_model)) pe[:, 0::2] = torch.sin(position * div_term) pe[:, 1::2] = torch.cos(position * div_term) self.register_buffer('pe', pe.unsqueeze(0)) def forward(self, x): # x: (batch, seq_len, d_model) return x + self.pe[:, :x.size(1)]

Transformer Block

class TransformerBlock(nn.Module): def __init__(self, d_model, num_heads, d_ff, dropout=0.1): super().__init__() self.attention = MultiHeadAttention(d_model, num_heads) self.norm1 = nn.LayerNorm(d_model) self.ffn = nn.Sequential( nn.Linear(d_model, d_ff), nn.ReLU(), nn.Linear(d_ff, d_model) ) self.norm2 = nn.LayerNorm(d_model) self.dropout1 = nn.Dropout(dropout) self.dropout2 = nn.Dropout(dropout) def forward(self, x, mask=None): # Self-attention attn = self.attention(x, x, x, mask) x = self.norm1(x + self.dropout1(attn)) # Feed-forward ffn = self.ffn(x) x = self.norm2(x + self.dropout2(ffn)) return x

Encoder

class TransformerEncoder(nn.Module): def __init__(self, vocab_size, d_model, num_heads, d_ff, num_layers, dropout=0.1): super().__init__() self.embedding = nn.Embedding(vocab_size, d_model) self.pe = PositionalEncoding(d_model) self.layers = nn.ModuleList([ TransformerBlock(d_model, num_heads, d_ff, dropout) for _ in range(num_layers) ]) self.dropout = nn.Dropout(dropout) def forward(self, x): # x: (batch, seq_len) - Token IDs x = self.embedding(x) # (batch, seq_len, d_model) x = self.pe(x) x = self.dropout(x) for layer in self.layers: x = layer(x) return x

# Hyperparameter vocab_size = 10000 d_model = 512 num_heads = 8 d_ff = 2048 num_layers = 6 batch_size = 32 # Modell model = TransformerEncoder(vocab_size, d_model, num_heads, d_ff, num_layers) # Dummy Input: (batch, seq_len) input_ids = torch.randint(0, vocab_size, (batch_size, 64)) # Forward Pass output = model(input_ids) # (batch, seq_len, d_model) # Loss (z.B. für Language Modeling) logits = linear_head(output) # (batch, seq_len, vocab_size) loss = cross_entropy_loss(logits, targets) # Backward loss.backward() optimizer.step()

Decoder (für Sequence-to-Sequence)

class TransformerDecoder(nn.Module): def __init__(self, vocab_size, d_model, num_heads, d_ff, num_layers, dropout=0.1): super().__init__() self.embedding = nn.Embedding(vocab_size, d_model) self.pe = PositionalEncoding(d_model) self.layers = nn.ModuleList([ TransformerBlock(d_model, num_heads, d_ff, dropout) for _ in range(num_layers) ]) # Cross-attention layer self.cross_attention = nn.ModuleList([ MultiHeadAttention(d_model, num_heads) for _ in range(num_layers) ]) self.output_proj = nn.Linear(d_model, vocab_size) def forward(self, x, encoder_output): x = self.embedding(x) x = self.pe(x) for i, layer in enumerate(self.layers): x = layer(x) # Cross-attention x = self.cross_attention[i](x, encoder_output, encoder_output) logits = self.output_proj(x) return logits
✨ Tipp: Das ist der Kern eines modernen LLMs! GPT ändert nur den Decoder (kein Encoder), BERT ändert nur den Encoder. T5 nutzt Encoder-Decoder wie hier.

🔟 Training & Real-World Use Cases

Transformers sind beeindruckend, aber wie trainiert man sie wirklich? Und wo werden sie verwendet?

Training ein großes Transformer-Modell

Daten-Vorbereitung

  • Tokenization: Text in Token IDs (z.B. mit BPE)
  • Sequenzen bilden: Fixed length (z.B. 512 Tokens)
  • Normalisierung: Lowercasing, Spezialzeichen behandeln
  • Batching: Padding/Masking für variable Längen

Loss Functions

Je nach Task:

  • Language Modeling: Cross-Entropy Loss auf nächstes Token
  • Classification: Cross-Entropy Loss auf Label
  • Machine Translation: Cross-Entropy Loss auf Target-Sequenz
  • Contrastive: SimCLR oder Triplet Loss
# Language Model Training Loop for epoch in range(num_epochs): for batch in train_loader: input_ids = batch['input_ids'] # (batch, seq_len) target_ids = batch['target_ids'] # (batch, seq_len) # Forward logits = model(input_ids) # (batch, seq_len, vocab_size) # Loss (vorhersage für jede Position) loss = cross_entropy( logits.view(-1, vocab_size), target_ids.view(-1) ) # Backward optimizer.zero_grad() loss.backward() torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0) optimizer.step()

Warum Transformers so effektiv sind

  • Parallelisierbarkeit: Trainiert schneller auf GPUs/TPUs als RNNs
  • Lange Kontexte: Kann lang-reichweite Abhängigkeiten lernen
  • Transfer Learning: Vortrainierte Modelle sind auf vielen Tasks sehr gut
  • Skalierbarkeit: Mehr Daten & Parameter = bessere Performance (Scaling Laws!)

Real-World Anwendungen

1. Language Models (GPT-Serie)

GPT, GPT-2, GPT-3, GPT-4, GPT-4 Turbo

Nur Decoder-basiert, trainiert auf "predict next token".
Größen: 125M (GPT-2) bis 1.7T+ (GPT-4)

Use Cases: Text-Generation, Conversation, Code-Generation, Summarization, Q&A

2. Bidirektionale Models (BERT, RoBERTa)

BERT - Bidirectional Encoder Representations from Transformers

Nur Encoder, trainiert mit Masked Language Modeling (MLM).
Größe: 110M (Base) bis 340M (Large)

Use Cases: Text Classification, Named Entity Recognition, Semantic Search, Paraphrase Detection

3. Sequence-to-Sequence (T5, mBART)

T5 - Text-to-Text Transfer Transformer

Encoder + Decoder, trainiert auf verschiedene NLP Tasks.
Größe: 60M bis 11B

Use Cases: Machine Translation, Text Summarization, Question Answering, Data Augmentation

4. Vision Transformers (ViT)

Vision Transformer - Transformer für Bilder!

Teile Bilder in Patches auf, nutze Standard Transformer-Architektur.

Use Cases: Image Classification, Object Detection, Image Retrieval

5. Multimodal Models (CLIP, LLaVA)

CLIP - Contrastive Language-Image Pre-training

Trainiere Text-Encoder & Image-Encoder gemeinsam.
Verstehe Beziehung zwischen Text und Bildern.

Use Cases: Image Search, Visual Understanding, Zero-shot Classification

Training Tipps & Tricks

Learning Rate Scheduling

# Warmup + Decay (wie in Transformer Paper) def get_learning_rate(step, d_model, warmup_steps=4000): return (d_model ** -0.5) * min( (step + 1) ** -0.5, (step + 1) * warmup_steps ** -1.5 ) # Oder: Cosine Annealing scheduler = torch.optim.lr_scheduler.CosineAnnealingLR( optimizer, T_max=num_epochs )

Gradient Accumulation (für große Batch Sizes)

accumulation_steps = 4 for i, batch in enumerate(train_loader): outputs = model(batch) loss = criterion(outputs, batch['labels']) loss.backward() if (i + 1) % accumulation_steps == 0: optimizer.step() optimizer.zero_grad()

Mixed Precision Training (schneller!)

from torch.cuda.amp import autocast, GradScaler scaler = GradScaler() for batch in train_loader: with autocast(): outputs = model(batch) loss = criterion(outputs, batch['labels']) scaler.scale(loss).backward() scaler.step(optimizer) scaler.update()

Skalierungsgesetze (Scaling Laws)

Empirische Beobachtung: Mit mehr Parametern und Daten wird das Modell exponentiell besser!

Loss ≈ A * (N ** -α) + B * (D ** -β)

Wo N=Netzwerk-Größe, D=Datenmenge, α≈0.07, β≈0.1

Das bedeutet: Größer = Besser! (In Maßen, natürlich.)

Deployment & Inference

KV-Cache (Key-Value Caching)

Im Decoder beim Inference: Speichere K und V aus vorigen Schritten!

# Ohne KV-Cache: O(n²) Speicher für n Tokens # Mit KV-Cache: O(n) Speicher # Im Decoder: past_key_values = None for i in range(max_tokens): logits, past_key_values = model( input_ids[:, -1:], # Nur letztes Token! past_key_values=past_key_values ) next_token = sample(logits) input_ids = torch.cat([input_ids, next_token], dim=-1)

Quantization (Model Compression)

Reduziere Modellgröße für schnelleres Inference:

  • INT8 Quantization: 4x kleiner
  • INT4 Quantization: 8x kleiner (bei etwas Qualitätsverlust)
  • Distillation: Trainiere kleineres Modell von großem
❓ Warum ist KV-Caching im Decoder wichtant?
A) Um zu vermeiden, dass alle vorherigen Tokens neu verarbeitet werden
B) Um Speicher zu sparen
C) Um die Gradientenberechnung zu beschleunigen
D) Um Fehler zu reduzieren
✅ Richtig! Im Decoder müssen wir beim nächsten Token wissen, was die Attention für die vorherigen Tokens war. KV-Cache speichert diese, um Re-computation zu vermeiden.
