1🎯Einführung: Was sind Transformer?

Transformer sind eine revolutionäre neuronale Netzwerk-Architektur, die 2017 im Paper "Attention Is All You Need" eingeführt wurde. Sie bilden die Grundlage moderner großer Sprachmodelle wie GPT, BERT und Claude. Der Schlüssel zum Erfolg: Sie verarbeiten Sequenzen parallel statt sequenziell, was sie viel schneller und skalierbarer macht als ihre Vorgänger.

Statt Information schrittweise durch eine Sequenz zu "laufen" (wie bei RNNs), schauen Transformer auf alle Positionen gleichzeitig. Jedes Token kann sich "selbst aufmerksam machen" auf alle anderen Tokens – das ist die Magie des Self-Attention Mechanismus.

Transformers bestehen aus zwei Hauptkomponenten: einem Encoder (versteht die Input-Sequenz) und einem Decoder (generiert die Output-Sequenz). Jeder besteht aus gestapelten Blöcken mit Attention- und Feed-Forward-Schichten.

🎓 Wichtigste Konzepte

  • Parallel Processing: Alle Tokens gleichzeitig verarbeitet, nicht sequenziell
  • Self-Attention: Jedes Token kann mit jedem anderen Token interagieren
  • Encoder-Decoder: Zwei symmetrische Komponenten für Input und Output
  • Skalierbarkeit: Funktioniert mit Milliarden von Parametern
❓ Warum sind Transformers so erfolgreich?

Weil sie drei Dinge perfekt kombinieren: (1) Sie können lange-Reichweiten-Abhängigkeiten modellieren, (2) Sie sind leicht zu parallelisieren, und (3) der Attention-Mechanismus ist interpretierbar – wir können sehen, worauf das Modell "schaut".

2⚠️Das Problem mit RNNs

Bevor Transformers kamen, dominierte eine andere Architektur: Recurrent Neural Networks (RNNs). Sie verarbeiten Sequenzen schrittweise: Token für Token, immer in Abhängigkeit vom vorherigen Hidden State. Das klingt logisch – aber es hat massive Probleme.

Das größte Problem: Vanishing Gradient Problem. Wenn du eine lange Sequenz zurückpropagierst, werden die Gradienten exponentiell kleiner. Nach 50-100 Schritten ist praktisch nichts mehr übrig – das Modell vergisst, was am Anfang passiert ist. RNNs können also nicht wirklich lange Abhängigkeiten lernen, obwohl das ihre Hypothese war.

Zweites Problem: Sequenzielles Processing. Ein RNN muss Token 1 verarbeiten, um Token 2 zu verarbeiten, um Token 3 zu verarbeiten. Das kann nicht parallelisiert werden. Bei einer Sequenz mit 1000 Tokens brauchst du 1000 sequenzielle Schritte – das ist langsam und uneffizient für moderne GPUs.

LSTMs und GRUs versuchen, das zu beheben, aber die fundamentalen Probleme bleiben. Mit Transformers und Attention ist alles anders.

# RNN: Sequenziell hidden = init_hidden() for token in sequence: hidden = rnn_step(token, hidden) # Muss sequenziell sein! output = hidden # Transformer: Parallel attention_output = attention(sequence, sequence, sequence) # Alle gleichzeitig!

🎓 RNN-Probleme zusammengefasst

  • Vanishing Gradients: Lange Abhängigkeiten können nicht gelernt werden
  • Keine Parallelisierung: Sequenzielles Processing ist langsam
  • Speicher-Overhead: Muss alle Hidden States speichern
  • Skalierung: Nicht praktisch für sehr lange Sequenzen

3👀Attention: Der Schlüsel

Attention ist das Konzept, das Transformers revolutionär macht. Die Idee ist einfach: Nicht alle Teile einer Sequenz sind gleich wichtig. Wenn du einen Satz liest, fokussierst du dich auf relevante Wörter und ignorierst unwichtige.

Formell: Attention berechnet, wie viel Gewicht jedes Token auf jedes andere Token legen sollte. Es ist ein Mechanismus, der automatisch wichtige Abhängigkeiten lernt. Wenn das Wort "Bank" in einem Satz vorkommt, kann Attention automatisch herausfinden, dass "Fluss" näher bedeutet als "Geldfluss".

Der Attention-Mechanismus hat drei Komponenten: Query (Q), Key (K), und Value (V). Die Query fragt "worauf sollte ich achten?", die Keys antworten "hier bin ich", und die Values sagen "hier ist meine Information". Das ist wie eine Datenbank-Abfrage: Du suchst (Q), findest Matches (K), und erhältst Ergebnisse (V).

Mit Attention können alle Tokens parallel auf alle anderen Tokens "schauen". Wenn Token A die Information von Token B braucht, berechnet der Attention-Mechanismus automatisch ein Gewicht. Keine feste Entfernung, keine Sequenzialisierung – nur reine Semantik.

Attention(Q, K, V) = softmax(Q·KT/√dk) · V
import torch import torch.nn.functional as F # Attention berechnen def attention(Q, K, V): # Scores: wie sehr sollte jedes Q auf jedes K achten? scores = torch.matmul(Q, K.transpose(-2, -1)) scores = scores / torch.sqrt(torch.tensor(Q.shape[-1])) # Normalisieren mit Softmax attention_weights = F.softmax(scores, dim=-1) # Gewichtete Summe der Values output = torch.matmul(attention_weights, V) return output, attention_weights
❓ Warum dividieren durch √d_k?

Die Skalierung sorgt dafür, dass die Scores nicht zu groß werden (was zu einem zu spitzen Softmax führt). d_k ist die Dimension der Keys – bei 64-dimensional Keys skalierst du mit √64 = 8.

4📏Scaled Dot-Product Attention

Scaled Dot-Product Attention ist die konkrete Implementierung des Attention-Mechanismus. Es ist elegant und effizient. Die "Dot-Product" Teil bedeutet, dass wir einfach das Skalarprodukt von Query und Key berechnen – das misst die Ähnlichkeit.

Der Prozess hat 4 Schritte: (1) Score berechnen: Q · KT – das sagt, wie sehr jedes Query zu jedem Key "passt". (2) Skalieren: Dividieren durch √d_k – dadurch werden die Gradienten stabiler. (3) Softmax: Das konvertiert Scores zu Wahrscheinlichkeiten (zwischen 0 und 1). (4) Gewichtete Summe: Multipliziere mit Values – das ist dein Ergebnis.

Warum ist das besser als andere Attention-Mechanismen? Erstens: Es ist computationally effizient – du kannst es mit Matrix-Multiplikationen implementieren, die GPUs lieben. Zweitens: Es funktioniert gut in der Praxis – alle modernen Transformers nutzen diese Methode.

k Skalieren Step 3 Softmax Gewichte Step 4 × V Output Matrix Dimensionen Q, K, V: (Seq, d_k) Scores: (Seq, Seq) Output: (Seq, d_k) Beispiel: Sequenz=10, d_k=64 Scores: 10×10 = 100 Attention-Gewichte ⚠️ Komplexität: O(n²) Für lange Sequenzen (n>10000) wird quadratisch teuer! Deswegen gibt es Optimierungen (Linear Attention, Sparse Attention)

🎓 Warum Scaled Dot-Product?

  • Effizient: Matrix-Multiplikation ist GPU-optimiert
  • Stabil: Skalierung verhindert explodierende/vanishing Gradienten
  • Interpretierbar: Attention-Gewichte zeigen, was das Modell "sieht"
  • Flexible Länge: Funktioniert mit beliebig langen Sequenzen
❓ Warum ist O(n²) ein Problem?

Bei n=1000 brauchst du 1 Million Attention-Berechnungen. Bei n=100000 (wie in Long Context LLMs) sind es 10 Milliarden – das ist langsam und speicherintensiv. Deswegen forschen Leute an Linear Attention und anderen Optimierungen.

5🎼Multi-Head Attention

Eine Single Attention-Head kann nur "eine Sache gleichzeitig" repräsentieren. Wenn du 10 verschiedene Beziehungen modellieren möchtest (z.B. "wer ist das Subjekt?", "wo ist das Verb?", "wer ist das Objekt?"), brauchst du mehrere Heads. Das ist die Idee hinter Multi-Head Attention.

Statt einer großen Attention-Matrix (d_model × d_model) teilst du sie in mehrere kleinere Matrizen auf. Mit 8 Heads und d_model=512 hat jeder Head 64 Dimensionen. Jeder Head lernt, sich auf verschiedene Aspekte zu konzentrieren. Head 1 könnte "syntaktische Abhängigkeiten" lernen, Head 2 "semantische Rollen", Head 3 "coreference resolution" usw.

Nach Attention-Berechnung in jedem Head kombinierst du die Outputs (concaten) und multiplizierst mit einer learned Weight Matrix. Das Ergebnis: Dein Modell kann gleichzeitig mehrere Beziehungen modellieren – viel ausdrucksstärker als ein einzelner Head.

class MultiHeadAttention(nn.Module): def __init__(self, d_model, num_heads): super().__init__() self.num_heads = num_heads self.d_k = d_model // num_heads # Linear Transformationen für alle Heads self.W_q = nn.Linear(d_model, d_model) # Q self.W_k = nn.Linear(d_model, d_model) # K self.W_v = nn.Linear(d_model, d_model) # V self.W_o = nn.Linear(d_model, d_model) # Output def forward(self, x): # x: (batch, seq_len, d_model) # Linear projections Q = self.W_q(x).view(batch, seq_len, self.num_heads, self.d_k) K = self.W_k(x).view(batch, seq_len, self.num_heads, self.d_k) V = self.W_v(x).view(batch, seq_len, self.num_heads, self.d_k) # Attention für jeden Head heads_output = [] for head in range(self.num_heads): head_out = scaled_dot_product_attention( Q[:, :, head, :], K[:, :, head, :], V[:, :, head, :] ) heads_output.append(head_out) # Concat + Linear concat = torch.cat(heads_output, dim=-1) output = self.W_o(concat) return output

🎓 Multi-Head Vorteile

  • Diverse Repräsentation: Verschiedene Heads lernen verschiedene Features
  • Robustheit: Fehler in einem Head werden von anderen kompensiert
  • Parallelisierung: Heads können parallel berechnet werden
  • Interpretierbarkeit: Du kannst schauen, was jeder Head lernt
❓ Wie viele Heads braucht man?

Standard: 8 Heads für BERT/GPT-2, 12-16 für größere Modelle. Es gibt kein magisches "beste" Zahl – zu viele Heads → overfitting, zu wenige → nicht genug Kapazität. Empirisch funktionieren 8-16 meistens gut.

6🧭Positional Encoding

Hier ist ein Problem: Wenn Transformer alle Tokens parallel verarbeiten, woher weiß das Modell, in welcher Reihenfolge sie sind? "Die Katze sitzt auf der Matte" ist nicht das gleiche wie "die Matte sitzt auf der Katze" – aber ohne Position würde Attention sie als gleich behandeln.

Die Lösung: Positional Encoding. Das ist ein mathematisches Signal, das die Position jedes Tokens kodiert. Anstatt dem Modell zu sagen "Token 0, Token 1, Token 2...", addierst du ein positionales Signal zu jedem Token-Embedding. Das Signal ist konstant – es ändert sich nicht während des Trainings.

Das Original-Transformer nutzt sinusoidale Positional Encodings. Die Formel sieht kompliziert aus, aber die Idee ist elegant: Höhere Frequenzen für nähere Positionen, niedrigere für entfernte. So kann das Modell relative Positionen leicht inferieren. Moderne Modelle nutzen manchmal gelernte oder rotations-basierte Positional Encodings (RoPE), aber sinusoidale funktionieren überraschend gut.

PE(pos, 2i) = sin(pos / 100002i/d_model)

PE(pos, 2i+1) = cos(pos / 100002i/d_model)
import math def positional_encoding(seq_len, d_model): """Sinusoidale positionale Encodings generieren""" pe = torch.zeros(seq_len, d_model) position = torch.arange(0, seq_len, dtype=torch.float).unsqueeze(1) # dimension_index = torch.arange(0, d_model, 2).float() div_term = torch.exp( torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model) ) pe[:, 0::2] = torch.sin(position * div_term) # Gerade Indizes pe[:, 1::2] = torch.cos(position * div_term) # Ungerade Indizes return pe

🎓 Wichtige Eigenschaften

  • Relative Positionen: Modell kann relative Abstände lernen
  • Keine Extrapolation nötig: Funktioniert auch für längere Sequenzen
  • Deterministic: Wird nicht trainiert – ist konstant
  • Komplementär zu Tokens: Wird zum Token-Embedding addiert
❓ Warum Sinus/Kosinus?

Die Sinuswelle ermöglicht es dem Modell, relative Positionen durch lineare Transformationen zu berechnen. sin(a+b) und cos(a+b) können aus sin(a), cos(a), sin(b), cos(b) berechnet werden – das ermöglicht Position-Awareness mit minimaler Komplexität.

7🏗️Der Transformer Block

Jetzt setzen wir es zusammen. Ein Transformer Block ist die Grundeinheit eines Transformers. Er kombiniert zwei Hauptkomponenten: Multi-Head Attention und ein Feed-Forward Network (FFN).

Der Prozess: (1) Eingabe wird mit Layer Normalization normalisiert. (2) Multi-Head Attention wird angewandt – das lässt Tokens sich "gegenseitig Aufmerksamkeit geben". (3) Residual Connection: Das Ergebnis wird zur Original-Eingabe addiert. Das verhindert, dass die Information verloren geht, besonders in tiefen Netzwerken. (4) Layer Norm erneut. (5) Ein FFN wird angewandt – zwei Linear Layers mit ReLU dazwischen, die Nicht-Linearität einführen. (6) Residual Connection erneut.

Das FFN hat typischerweise die Form: Linear(d_model → 4×d_model) → ReLU → Linear(4×d_model → d_model). Die "4x" ist eine empirische Konvention – macht das Netzwerk breiter, ermöglicht mehr nicht-lineare Expressiveness.

class TransformerBlock(nn.Module): def __init__(self, d_model, num_heads, d_ff): super().__init__() self.attention = MultiHeadAttention(d_model, num_heads) self.ffn = nn.Sequential( nn.Linear(d_model, d_ff), nn.ReLU(), nn.Linear(d_ff, d_model) ) self.norm1 = nn.LayerNorm(d_model) self.norm2 = nn.LayerNorm(d_model) def forward(self, x): # Self-Attention mit Residual attn_out = self.attention(self.norm1(x)) x = x + attn_out # Residual Connection # FFN mit Residual ffn_out = self.ffn(self.norm2(x)) x = x + ffn_out # Residual Connection return x

🎓 Block-Komponenten erklärt

  • Layer Norm: Stabilisiert das Training – normalisiert pro Sample
  • Residual Connections: Gradient kann direkt fließen – hilft tiefe Netze
  • Multi-Head Attention: Lernt komplexe Abhängigkeiten
  • FFN: Führt nicht-lineare Transformationen ein – eine Art "Lookup-Table"
❓ Warum ist d_ff = 4 × d_model standard?

Das ist empirisch. Ein kleineres FFN wäre nicht ausdrucksstark genug, ein größeres verschwendet Parameter. 4x ist ein guter Kompromiss – hat sich in GPT, BERT und anderen Modellen bewährt.

8🎭Die Complete Transformer Architektur

Jetzt haben wir alle Teile. Ein Complete Transformer bestellt aus dem Encoder und Decoder. Der Encoder verarbeitet die Input-Sequenz und erzeugt eine Repräsentation. Der Decoder nimmt diese Repräsentation und generiert die Output-Sequenz Token für Token.

Der Encoder: N gestapelte Transformer Blocks. Jeder Block macht Self-Attention (jedes Token schaut alle Tokens an). Das Ergebnis ist eine "verstanden" Repräsentation der Input-Sequenz. Der Encoder ist völlig parallel – alle Tokens können gleichzeitig verarbeitet werden.

Der Decoder: Auch N gestapelte Blocks, aber mit einer Modifikation. Die Self-Attention ist Masked – das Modell kann nur vorherige Tokens anschauen, nicht zukünftige. (Das heißt "Causal Attention".) Zusätzlich hat jeder Block Cross-Attention – das ist Attention zwischen Decoder-Tokens (Query) und Encoder-Output (Key/Value). So kann der Decoder von der Encoder-Repräsentation profitieren.

Modern LLMs (GPT, Claude) nutzen nur den Decoder – ohne Encoder. Das ist einfacher und funktioniert überraschend gut für viele Aufgaben.

class Transformer(nn.Module): def __init__(self, d_model, num_heads, num_blocks, d_ff, vocab_size): super().__init__() # Encoder self.encoder_embedding = nn.Embedding(vocab_size, d_model) self.encoder_blocks = nn.ModuleList([ TransformerBlock(d_model, num_heads, d_ff) for _ in range(num_blocks) ]) # Decoder self.decoder_embedding = nn.Embedding(vocab_size, d_model) self.decoder_blocks = nn.ModuleList([ TransformerBlock(d_model, num_heads, d_ff) for _ in range(num_blocks) ]) self.decoder_cross_attn = nn.ModuleList([ MultiHeadAttention(d_model, num_heads) for _ in range(num_blocks) ]) # Output self.output_linear = nn.Linear(d_model, vocab_size) def forward(self, src, tgt): # Encoder enc_out = self.encoder_embedding(src) for block in self.encoder_blocks: enc_out = block(enc_out) # Decoder dec_out = self.decoder_embedding(tgt) for i, block in enumerate(self.decoder_blocks): dec_out = block(dec_out) # Masked self-attention # Cross-attention: Decoder attends to Encoder dec_out = dec_out + self.decoder_cross_attn[i]( query=dec_out, key=enc_out, value=enc_out ) # Output projection logits = self.output_linear(dec_out) return logits

🎓 Komponenten zusammengefasst

  • Encoder: Verarbeitet Input parallel, erzeugt Kontext
  • Decoder: Generiert Output sequenziell, nutzt Encoder-Kontext
  • Masked Attention: Verhindert, dass das Modell "voraus schaut"
  • Cross-Attention: Verbindung zwischen Encoder und Decoder
  • Output Layer: Gibt Wahrscheinlichkeiten für nächstes Token
❓ Was ist "Decoder-Only"?

GPT und andere moderne LLMs verwenden nur den Decoder – ohne Encoder. Der Grund: Für viele Aufgaben (Spracherzeugung, Code-Generierung) brauchst du den Encoder nicht. Decoder-Only ist einfacher und genauso ausdrucksstark. Du hängst einfach vorherige Tokens als Context an – das ist alles, was du brauchst.

🎉Zusammenfassung: Alles zusammen

🧠 Was Transformer Gutes können:

  • ✓ Lange Abhängigkeiten modellieren
  • ✓ Parallel verarbeiten (schnell!)
  • ✓ Multi-Head erlaubt vielfältige Muster
  • ✓ Skaliert zu Milliarden von Parametern

⚠️ Limitationen:

  • ✗ Attention ist O(n²) – teuer für lange Sequenzen
  • ✗ Braucht viel Daten und Compute zum Trainieren
  • ✗ Versteht keine "physikalischen" Gesetze von selbst
  • ✗ Positional Encoding ist nicht perfekt
